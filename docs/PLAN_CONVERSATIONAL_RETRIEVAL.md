# Plan — Your Library Knowledge Base (+ web search and conversations) — Python, full rebuild

Status: **locked, not yet implemented**
Scope: a **complete standalone Python system built from scratch** (`agentic-ai-py/`).
Python port + full rebuild of the Go system in `~/golang-projects/agentic-ai/`.

> **Feature parity:** this is the Python counterpart of `~/golang-projects/agentic-ai/PLAN_CONVERSATIONAL_RETRIEVAL.md`. Both plans expose the **same user-facing features** — Web / My Library toggles, conversation memory, `recall_history`, past-conversations sidebar, clarify-back, tiered context management, rolling summary, modern Lucide icons, JSON-now/SSE-later. Both now **own PDF search + ingestion in-repo** (talk to Pinecone directly) with a Mongo **`books` catalog** powering book-title narrowing + clarify-back. The differences are **implementation only**: (1) language; (2) this repo builds it from scratch, whereas the Go repo lifts Omni-RAG's Go code (`pinecone.go`, `embedder.go`, `ingestion/main.go`); (3) this repo uses a python.org theme, the Go repo keeps its dark/emerald theme. If a feature changes in one plan, change it in the other.

### What "full rebuild" means
Unlike the Go repo (which reused an external Pinecone retrieval service and only had the agent + Go MCP server), **everything is built in Python here**:
- **PDF ingestion** — upload, extract, chunk, embed, upsert to Pinecone. *(new, Python)*
- **PDF retrieval** — query → embed → Pinecone search → SSE. Replaces the external `:8081`. *(new, Python)*
- **MCP server** — Mongo CRUD tools over SSE. Replaces the Go `:8083`. *(rebuilt, Python)*
- **Agent** — conversational ReAct + memory + source toggles. *(new, Python)*

External dependencies remain: **MongoDB**, **Pinecone**, **Ollama** (LLM + embeddings), **Anthropic** (optional), **Tavily**.

## 1. Goal

A **conversation-style assistant** where the user explicitly chooses where answers come from, via two toggle buttons:

| Button | Effect when ON |
|---|---|
| 🌐 **Web** | Model may use `web_search` (Tavily) |
| 📚 **My Library** | Model may use internal sources: `search_pdf` (Pinecone book vectors) **and** the Mongo MCP CRUD tools |

| Web | My Library | Behaviour |
|---|---|---|
| off | off | **No retrieval tools.** Model answers from its own training only. If unsure, suggests enabling 🌐 Web. |
| off | on | Internal only — Pinecone PDF search + Mongo MCP. No web. |
| on | off | Web only — Tavily. No internal sources. |
| on | on | Use **both** internal + web, merge into the answer. |

Plus:
- **Conversation memory** — references earlier turns ("what did I ask first", "summarize our chat").
- **Past-conversations sidebar** — click to reopen any prior conversation.
- **Clarify-back** — when My Library is on but a topic can't be pinned to a specific book (thousands of books dilute vector accuracy), the model asks *"do you mean *Game of Thrones* or {X}?"* instead of guessing. (Backed by per-book metadata stored at ingestion — see §4.)
- **Everyday queries** — work fine with both toggles off (plain LLM).

## 2. Key architectural decisions

1. **Vector search = Pinecone**, with **embeddings generated by Ollama** (same approach the Go/Omni-RAG stack used). Embedding model + dimension must match the Pinecone index dimension.
2. **"My Library" = internal sources** = `search_pdf` (Pinecone) + Mongo MCP CRUD.
3. **Conversation history is stored in MongoDB via a direct async driver (`motor`) in the agent — NOT via the MCP tool.** Persistence is deterministic app plumbing; the MCP tool is the *model's* tool and includes destructive ops (`delete_document`). Same Mongo instance, separate access layer.
4. **Context management = token-budgeted tiered memory** (§7), not "resend everything."
5. **Deep recall = a dedicated read-only `recall_history` tool**, conversation-scoped — never the general MCP CRUD.
6. **Book-title narrowing via a Mongo `books` catalog + clarify-back** — ingestion writes `book_title` + a stable `source_file_id` into Pinecone metadata AND a Mongo `books` catalog (title/aliases ↔ `source_file_id`). At query time the agent resolves a spoken title against the catalog → one match filters the vector search to that book; ambiguous → clarify-back; none → search all. Reduces cross-book dilution.
7. **Response mode = simple JSON for now.** SSE streaming of the agent answer deferred (`todo.md`); the **retrieval** service does stream SSE.

## 4. `retrieval/` — ingestion + search (Python, replaces :8081)

### 4.1 Ingestion — `POST /api/ingest` (multipart PDF upload)
Pipeline (`ingest.py`):
1. **Extract** text per page (`pdf.py`, pypdf).
2. **Chunk** with overlap (`chunking.py`).
3. **Embed** each chunk via Ollama (`embeddings.py`, `POST {OLLAMA_HOST}/api/embeddings`).
4. **Upsert** to Pinecone (`pinecone_store.py`) with metadata (schema aligned to the existing `omni-rag-pdfs` index + `book_title`):
   ```jsonc
   { "text_content": "...", "book_title": "<from filename/form>", "source_file_id": "<stable book id>",
     "page_number": 12, "chapter": "...", "chunk_index": 3 }
   ```
   `book_title` powers per-book identification; `source_file_id` is the stable id used for vector filtering.
5. **Upsert the `books` catalog** in Mongo: `{ "_id": "<source_file_id>", "title": "...", "aliases": [...], "chunk_count": N }`.
- Returns ingestion summary (chunks upserted, book_title, source_file_id).
- Optional simple **upload UI** under `retrieval/app/static/`.

> Existing title-less vectors in `omni-rag-pdfs` are handled by **re-ingesting** to populate `book_title` + the catalog (user-chosen).

### 4.2 Search — `POST /api/search` `{ "query": "..." }` → **SSE**
Mirror the documented SSE contract so the agent's `pdf_search` client is portable:
```
event: status   data: {"stage":"embedding","message":"Generating query embedding via Ollama…"}
event: status   data: {"stage":"retrieval","message":"Querying Pinecone for top K matches…"}
event: sources  data: {"sources":[{"text_content":"...","book_title":"...","source_file_id":"...","page_number":12,"score":0.82}, ...]}
event: error    data: {"stage":"retrieval","message":"..."}    // when no matches / failure
event: done     data: {}
```
- `search.py`: embed query → Pinecone `query(topK, includeMetadata=True)` → emit `sources` (or `error` when empty).
- Optional `book` param: resolved against the Mongo `books` catalog (case-insensitive / alias / fuzzy) → 1 match filters by its `source_file_id`; multiple → return candidates for clarify-back; 0 → unfiltered.

## 6. `agent/` — conversational ReAct agent (:8082)

### 6.1 Data model (MongoDB via `motor`, written only by the agent)
`conversations`: `{ _id, title, created_at, updated_at, summary, summary_upto_seq }`
`messages`: `{ _id, conversation_id, seq, role: user|assistant, content, sources?, created_at }`
> `content` = **final text only**; raw tool observations (Pinecone chunks/web text) are **never** stored or replayed.

### 6.2 `store.py` (motor)
`create_conversation`, `append_message`, `get_recent_messages(k)`, `get_messages_up_to/after_seq`, `get_all_user_messages`, `get_first_n_user_messages`, `search_messages`, `get_conversation`, `update_summary`, `list_conversations`, `get_conversation_messages`.

### 6.3 `models.py`
```python
class AgentRequest(BaseModel):
    query: str
    conversation_id: str | None = None
    use_web: bool = False
    use_library: bool = False

class AgentResponse(BaseModel):
    answer: str
    conversation_id: str
    title: str
    needs_clarification: bool = False
```

### 6.4 `tools/`
- `base.py` — `Tool` protocol (`name`, `schema()`, `execute()`), `Manager` compiling schemas into the prompt section.
- `pdf_search.py` — async httpx SSE client to `:8081/api/search`; collect `event: sources`, treat `event: error` as empty; return `[PDF_SUCCESS|...]` / `[PDF_EMPTY|...]` markers.
- `web_search.py` — Tavily, returns `answer`.
- `mcp_tool.py` — MCP **client** (Python SDK, SSE to :8083): `action:"list_tools"` → discover, else call.
- `recall_history.py` — **read-only**, conversation-scoped (backend injects conv_id); `mode: all|first_n|search`.

### 6.5 `react.py`
`async run_react(conv_id, query, use_web, use_library) -> (answer, needs_clarification)`:
1. **Per-request tool set:** always `recall_history`; `use_library` → `search_pdf` + `mcp`; `use_web` → `web_search`.
2. **Budgeted prompt** (§7): system (branched by toggles) + rolling summary + last K turns verbatim + current query.
3. **ReAct loop** (Thought/Action/Action Input/Observation, max `MAX_STEPS`); observations transient — only Final Answer persisted.
4. **Clarify-back:** `Clarification: ...` treated like `Final Answer:` → exit, `needs_clarification=True`.
5. **Prompt branches:** no tools → answer from own knowledge, suggest Web if unsure; personal on → "if a topic can't be pinned to a specific book, ask which book rather than guessing" (uses `book_title` in sources).

### 6.6 `main.py` endpoints
- `POST /api/agent/query` → create/lookup conversation, persist user msg, `run_react`, persist assistant msg, maybe summarize, return `AgentResponse`.
- `GET /api/conversations` (sidebar) · `GET /api/conversations/{id}` (reopen) · `DELETE /api/conversations/{id}` (optional).
- Serve `agent/app/static/`. Startup: load config, pick LLM backend, register tools, connect MCP client.

## 7. Context management

Tiered, token-budgeted memory — never resend raw observations.

| Part | Size | Strategy |
|---|---|---|
| **User queries** | tiny | Recent verbatim; older reachable on demand via `recall_history`. |
| **Assistant answers** | medium | Last K turns verbatim; older folded into rolling `summary`. |
| **Tool observations** (Pinecone/web) | huge | **Never persisted, never replayed.** ← biggest win. |

Per-turn prompt = system (branched) + rolling summary + last K turns verbatim + current query + [transient observations].
Mechanism: `len/4` token-budget heuristic (fill newest-first); roll older-than-K into `conversations.summary` when `> SUMMARY_TRIGGER_TURNS`; full history always in Mongo; `recall_history` gives **exact** deep recall.

### vs ChatGPT/Claude
They resend the whole transcript (huge windows) and summarize only on overflow. Local Ollama can't assume that, so we budget + summarize earlier and add `recall_history` to keep prompts cheap. Cross-chat "Memory" is out of scope.

## 8. Tool/permission safety
- `recall_history` read-only + conversation-scoped (backend injects conv_id).
- MCP CRUD (`delete_document` etc.) only exposed when **My Library** is on; never used for conversation history.
- Conversation collections written **only** by the agent's `motor` driver, never by the model.

### 9.2 Icons — modern SVG icon set (no emoji)
**No emoji in the actual UI.** Use a modern, consistent **stroke-based SVG icon set — [Lucide](https://lucide.dev)** (MIT, the modern successor to Feather; huge set, crisp at small sizes). The emoji in this doc (🌐, 📚, …) are only shorthand for the Lucide icons below.

- **Delivery:** inline `<svg>` (preferred, themeable via `currentColor`, no extra request) or the Lucide CDN. Default size **18px**, stroke-width **2**, color `currentColor` so icons inherit the python.org blue/yellow/dark per context.
- **Mapping:**

| UI element | Lucide icon |
|---|---|
| Web toggle | `globe` |
| My Library toggle | `library` (alt: `book-open`) |
| Send / Ask (primary button) | `arrow-up` (alt: `send`) |
| New chat | `square-pen` (alt: `plus`) |
| Past-conversation item | `message-square` |
| Delete conversation | `trash-2` |
| Clarification turn | `help-circle` |
| Web source citation | `external-link` |
| Per-book source tag | `book` |
| Ingest / upload PDF | `upload` (alt: `file-up`) |
| Loading / reasoning | `loader-circle` (animate-spin) |
| Error banner | `circle-alert` |

> Icons inherit color from context: blue (`--py-blue`) for inactive/neutral, white on the dark top bar, `#333` on yellow buttons, yellow (`--py-yellow`) for accent marks. Keep stroke style uniform — don't mix Lucide with filled/emoji glyphs.
